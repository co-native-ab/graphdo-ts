# ADR-0002: Frontend Refactoring Approach

## Status

**Proposed**

## Context

graphdo-ts serves browser-facing HTML pages during interactive flows — MSAL login loopback and a browser-based option picker. These pages are generated by TypeScript functions in `src/templates/` that return complete HTML strings with inline `<style>` and `<script>` blocks, served by local HTTP servers (`src/loopback.ts` and `src/picker.ts`).

The current template implementation has three files:

- **`src/templates/styles.ts`** — exports a shared `BASE_STYLE` CSS string and an `escapeHtml()` utility function.
- **`src/templates/login.ts`** — three HTML page generators (`landingPageHtml`, `successPageHtml`, `errorPageHtml`) for the MSAL login loopback flow, each with its own inline `<style>` and `<script>` blocks.
- **`src/templates/picker.ts`** — an HTML page generator (`pickerPageHtml`) for the browser-based option picker, with its own inline `<style>` and `<script>` block.

Key forces at play:

- **Hardcoded color values**: Brand colors (Microsoft blue `#0078d4`, success green `#107c10`, error red `#d13438`) and other design values (background `#f5f5f5`, text `#333`, borders `#ddd`) are duplicated as raw hex strings across multiple template files. Changing the brand color requires finding and updating every occurrence.
- **No design system**: There is no single source of truth for colors, typography, spacing, or other design tokens. Each page defines its own inline styles independently, leading to subtle inconsistencies.
- **Inline JavaScript duplication**: The countdown timer logic (decrement counter, attempt `window.close()`, show manual-close fallback) is duplicated verbatim in `login.ts` (`successPageHtml`) and `picker.ts` (`pickerPageHtml`). Any bug fix or behavior change must be applied in both places.
- **Testability gap**: The inline CSS and JavaScript embedded within HTML template strings cannot be unit tested independently — they can only be validated by parsing the rendered HTML output.
- **Brand identity**: The project uses Microsoft blue (`#0078d4`) as the primary brand color, but graphdo-ts is a Co-native product with its own brand identity. The primary brand color should be `#70638c` (purple/violet, extracted from the project's logo assets).
- **Architecture constraint**: The local HTTP servers are minimal, serving a single HTML response per request. They are not designed to serve static assets (CSS files, JS files) from disk.

## Decision

Refactor the `src/templates/` module to separate concerns within the existing TypeScript-functions-returning-HTML-strings architecture. The overall pattern of inline HTML served by local HTTP servers remains unchanged; the refactoring improves internal structure, testability, and brand consistency.

### 1. Design Tokens as TypeScript Constants

Define all brand colors, typography, spacing, and other design tokens as typed constants in a single `src/templates/tokens.ts` file. CSS custom properties are generated from these constants and injected into a `:root` block within each page's `<style>` element. This keeps the single source of truth within the TypeScript ecosystem — no CSS files, preprocessors, or external tooling needed.

The brand palette is:

- Primary: `#70638c` (Co-native purple/violet, extracted from logo)
- Primary hover: `#5d5275` (darker shade)
- Primary active: `#4a4260` (darkest shade)
- Success: `#107c10` (standard green, unchanged)
- Error: `#d13438` (standard red, unchanged)
- Background: `#f5f5f5` (unchanged)
- Surface/card: `#ffffff`
- Text primary: `#333333`
- Text secondary: `#666666`
- Text muted: `#999999`
- Border: `#dddddd`

### 2. Separation of Concerns Within TypeScript

Restructure `src/templates/` into focused modules:

- **`tokens.ts`** — design token constants (colors, fonts, spacing, radii, shadows) as a typed object. Exports both the raw values and a function to generate CSS custom properties.
- **`styles.ts`** — CSS generation functions that consume tokens via CSS custom properties. Shared base styles plus page-specific style functions all reference `var(--token-name)` rather than hardcoded hex values.
- **`scripts.ts`** — extracted inline JavaScript as string constants or template functions. The countdown timer logic and picker selection logic are defined once and imported where needed.
- **`login.ts`** and **`picker.ts`** — HTML structure only, composing the page by importing styles, scripts, and tokens.

### 3. Brand Color Update

Replace Microsoft blue (`#0078d4`) with Co-native purple (`#70638c`) as the primary brand color across all browser-facing pages. Hover and active states derive from the primary color. All other visual elements (layout, spacing, typography, card style, success/error colors) remain identical.

### 4. No New Runtime Dependencies

All templates remain plain TypeScript functions returning HTML strings with inline `<style>` and `<script>` blocks. No CSS preprocessors, CSS-in-JS libraries, bundled static assets, or build step changes are introduced.

### 5. Testing Strategy

Leverage the existing vitest setup to add targeted tests:

- Unit tests for design token values (snapshot assertions on the token object).
- Unit tests for CSS generation functions (verify output contains expected custom properties and values).
- Unit tests for extracted JavaScript strings (countdown timer, picker selection logic).
- Template rendering tests to verify HTML structure includes token-derived CSS custom properties.
- Existing integration tests (`loopback.test.ts`, `picker.test.ts`) remain unchanged — they validate the server behavior and HTML response, which is unaffected by the internal refactoring.

## Consequences

### Positive

- **POS-001**: Single source of truth for brand colors and design tokens — changing the primary color or any design value requires editing one constant in `tokens.ts`, and all pages update automatically.
- **POS-002**: Extracted JavaScript modules eliminate code duplication — the countdown timer logic is defined once in `scripts.ts` and imported by both `login.ts` and `picker.ts`, so bug fixes apply everywhere.
- **POS-003**: Testable architecture — tokens, CSS generation functions, and script strings are all importable TypeScript constants that can be unit tested independently, without parsing rendered HTML.
- **POS-004**: No new dependencies or build complexity — the refactoring stays within the existing TypeScript + esbuild single-file bundle architecture, adding zero runtime dependencies.
- **POS-005**: Consistent brand identity — all browser-facing pages use the Co-native purple (`#70638c`) instead of Microsoft blue, establishing a distinct visual identity for the product.

### Negative

- **NEG-001**: CSS is still inline in the HTML output — while now generated from tokens rather than hardcoded, the rendered pages still contain `<style>` blocks rather than external stylesheets. This is a deliberate trade-off for the local HTTP server architecture.
- **NEG-002**: JavaScript is still inline in the HTML output — while extracted to a separate module for testability and deduplication, the rendered pages still contain `<script>` blocks. External script files would require static file serving that the servers do not support.
- **NEG-003**: Token changes require TypeScript compilation — updating a design token value means recompiling and rebuilding. This is consistent with the project's existing workflow (all changes require compilation) and not an additional burden.
- **NEG-004**: The refactoring adds more files to `src/templates/` (`tokens.ts`, `scripts.ts`, and a restructured `styles.ts`) — developers must understand the module structure to make template changes, though the separation makes each file simpler and more focused.
- **NEG-005**: CSS custom properties add a layer of indirection — styles reference `var(--color-primary)` rather than `#70638c` directly. This is the standard trade-off for any design token system, and the indirection is the mechanism that enables the single-source-of-truth benefit.

## Alternatives Considered

### External CSS/JS Files Served by the HTTP Servers

- **ALT-001**: **Description**: Extract CSS into `.css` files and JavaScript into `.js` files, then serve them as static assets from the local HTTP servers alongside the HTML pages.
- **ALT-002**: **Rejection Reason**: The local HTTP servers (`src/loopback.ts` and `src/picker.ts`) are minimal, designed for single-page responses. Adding static file serving (route handling for `/styles.css`, `/scripts.js`, MIME types, file reading) would significantly increase server complexity for pages that are small and self-contained. The additional HTTP round-trips would also increase page load time for no user-visible benefit.

### CSS-in-JS Library (e.g., Emotion, Styled-Components)

- **ALT-003**: **Description**: Use a CSS-in-JS library to define styles as JavaScript objects, generating CSS at runtime with scoped class names, theming support, and a design token system built into the library.
- **ALT-004**: **Rejection Reason**: These libraries are designed for client-side React/SPA applications, not server-rendered HTML strings. Adding a CSS-in-JS runtime would introduce unnecessary dependencies, increase the bundle size, and add complexity to a system that generates static HTML pages. The project's approach (TypeScript constants producing CSS strings) achieves the same design token benefits without any runtime library.

### CSS Files Bundled by esbuild

- **ALT-005**: **Description**: Create `.css` source files for styles, configure esbuild to bundle them alongside the TypeScript code, and embed the bundled CSS into the HTML templates at build time.
- **ALT-006**: **Rejection Reason**: The esbuild configuration bundles for Node.js (`platform: 'node'`), not for the browser. Adding CSS bundling would require a separate build step or a dual-platform build configuration. The CSS would still end up as inline strings in the HTML output, so the user-facing result would be identical — but with additional build complexity and a separate source format (CSS files vs. TypeScript constants) that fragments the design token system.

## Implementation Notes

- **IMP-001**: The `tokens.ts` module exports a typed `DESIGN_TOKENS` object with nested namespaces (e.g., `DESIGN_TOKENS.color.primary`, `DESIGN_TOKENS.spacing.cardPadding`) and a `cssCustomProperties()` function that returns a `:root { --color-primary: #70638c; ... }` string for injection into `<style>` blocks.
- **IMP-002**: The `scripts.ts` module exports JavaScript as template literal strings (e.g., `countdownScript(seconds: number): string`). These strings are injected into `<script>` blocks by the page generators. The functions accept parameters (countdown duration, selectors) to avoid hardcoding DOM IDs.
- **IMP-003**: The migration is internal to `src/templates/` — the public API (exported functions `landingPageHtml`, `successPageHtml`, `errorPageHtml`, `pickerPageHtml`, `escapeHtml`) remains unchanged. No changes are needed in `src/loopback.ts`, `src/picker.ts`, or any other consuming module.
- **IMP-004**: The only user-visible change is the primary brand color shifting from Microsoft blue (`#0078d4`) to Co-native purple (`#70638c`). Layout, spacing, typography, and interaction patterns remain identical.
- **IMP-005**: Success criteria: all existing tests pass without modification, new unit tests cover token values and CSS generation, and the rendered HTML output is visually identical except for the brand color change.
